# T-002: Database Schema — WhatsApp Tables (wa.*)

`wayfinder:task` | Parent: map.md

## Question

Produce the complete DDL for the `wa` Postgres schema. These tables absorb WassFlow's domain (sequences, triggers, inbox, campaigns, send queue) but redesigned for multi-tenancy with organization_id scoping. This schema handles WhatsApp automation/chabot features independently from the PostManager schema.

## Scope

Must cover these tables (absorbed and redesigned from WassFlow):

- **wa_instances** — id, organization_id, name, provider_type, api_key, session_id, webhook_url, status (enum: active, disconnected, pairing), qr_code_data, last_activity_at, created_at
- **wa_sequences** — id, organization_id, name, description, created_at
- **wa_sequence_steps** — id, sequence_id (FK), position (int), message_type (enum: text, image, video, audio, document), message_body, media_url, delay_seconds, created_at
- **wa_triggers** — id, organization_id, instance_id (FK), keyword, match_type (enum: exact, contains, starts_with), sequence_id (FK), enabled (bool), created_at
- **wa_messages** — id, organization_id, instance_id (FK), sender_jid, message_body, matched_keyword, trigger_id (FK, nullable), direction (enum: inbound, outbound), status (enum: received, processing, sent, failed), send_job_id (FK, nullable), created_at
- **wa_send_jobs** — id, organization_id, message_id (FK), sequence_id (FK), current_step (int), total_steps (int), status (enum: running, completed, failed), error_message, completed_at, created_at
- **wa_campaigns** — id, organization_id, name, campaign_type (enum: bulk_distribution, broadcast), instance_id (FK), group_list_id (FK), product_ids (UUID array), delay_min_seconds, delay_max_seconds, wave_delay_min_seconds, wave_delay_max_seconds, scheduling_mode (enum: automatic, manual_waves), wave_start_times (JSONB), scheduled_start_at, start_jitter_seconds, status (enum: draft, scheduled, running, paused, completed, cancelled, failed), total_events, completed_events, failed_events, completed_at, created_at
- **wa_campaign_events** — id, campaign_id (FK), product_id, group_jid, batch_index (int), send_order (int), scheduled_at, status (enum: pending, queued, sending, sent, failed), actual_sent_at, api_status_code, api_response, error_message, created_at
- **wa_send_queue** — id, session_id (FK → wa_instances), instance_api_key, recipient (jid), payload (JSONB), priority (int), attempts (int), max_attempts (int), status (enum: pending, processing, sent, failed), scheduled_at, executed_at, error_message, campaign_event_id (FK, nullable), message_id (FK, nullable), send_job_id (FK, nullable), presence_type, presence_duration_seconds, created_at
- **wa_group_lists** — id, organization_id, name, description, created_at
- **wa_group_list_items** — id, group_list_id (FK), group_jid, group_name

## Requirements

1. Every table has `organization_id` and RLS enabled
2. All timestamps use `timestamptz`, all IDs use `uuid`
3. WassFlow's `wf_*` naming changed to `wa_*` (WhatsApp Automation)
4. All WassFlow's single-tenant assumptions removed — org_id scoping added
5. Composite indexes matching WassFlow's hot query patterns plus org_id prefix
6. The `wa_send_queue` table is the PostingProvider target — PostManager's automated dispatch inserts rows here
7. The `wa_send_queue` and `wa_campaign_events` tables are candidates for partitioning (range on `scheduled_at`, monthly)
8. `wa_instances` corresponds to WA Posters in PostManager — they share the same concept but different schemas
9. Foreign keys with appropriate ON DELETE behavior

## Relationship to PostManager Schema

The `wa_send_queue` table is the handshake point between schemas:
- PostManager's automated campaign engine inserts into `wa.send_queue` via a Postgres function `pm.enqueue_dispatch()`
- WhatsApp automation reads from `wa.send_queue`, processes, and updates status
- Alternatively, if Postgres function coupling is too tight, the connection is via a shared Supabase Edge Function
