# T-003: PostingProvider Interface Contract

`wayfinder:grilling` | Parent: map.md

## Question

Define the exact `PostingProvider` interface, its three built-in implementations, and the data contract each one fulfills. This is the core pluggability boundary — every posting method must conform to this spec.

## Scope

1. **Interface definition** — exact TypeScript interface with method signatures, parameter types, return types, error types
2. **ManualProvider** — how does a Human Poster interact? What's the UI flow? What data does the provider expose (dispatch queue, "Mark as Done" action, actual_sent_at capture, late/missed detection)?
3. **AutomatedProvider** — how does it connect to WA Posters? Does it insert directly into `wa.send_queue`? Does it call a Supabase Edge Function? Does it call the WatsSender HTTP client directly?
4. **ExportProvider** — ZIP structure specification exactly: file naming convention, `captions.txt` format, `whatsapp_links.txt` format, `schedule.json` schema, media file ordering
5. **Provider registry** — how are providers registered and selected per Campaign? Single static registry or dynamic plugin system?
6. **Error contract** — what errors can each provider return? How does PostManager's dispatch engine handle provider failures?
7. **Status reporting contract** — how does each provider report back dispatch status? What statuses are valid?
8. **Configuration** — what per-Campaign configuration does each provider need?

## Context

From CONTEXT.md: "Three built-in implementations: Manual (Human Poster copies/pastes), Automated (WA Poster sends via API), and Export (downloadable ZIP with media + captions + links)."

From scheduling-engine-analysis.md: The dispatch engine loops, resolves variant + variable, checks anti-detection constraints, then calls `provider.send(dispatch)`. The provider returns `{ status, messageId?, error? }`.
