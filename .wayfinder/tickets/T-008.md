# T-008: WatsSender HTTP Client Specification

`wayfinder:task` | Parent: map.md

## Question

Produce the specification for the WatsSender HTTP client — the extracted, standalone module that wraps the WatsSender REST API. This module becomes the `AutomatedProvider`'s transport layer and is also used by WhatsApp automation (wa schema) for sending messages. Based on WassFlow's `src/lib/wasender.ts` and `src/lib/whatsapp/client.ts`.

## Scope

1. **Client initialization** — constructor params (apiKey, baseUrl, sessionId), config options (timeout, retry policy)
2. **Message sending methods** — sendText, sendImage, sendVideo, sendAudio, sendDocument, sendGroupedMedia — exact param types and return types
3. **Session management** — getSessions, getSessionStatus, initiatePairing, getQR
4. **Group management** — getGroups, getGroupMetadata, getGroupParticipants
5. **Presence** — sendPresence (typing indicator)
6. **Media** — decryptMedia (for incoming messages)
7. **Rate limiting** — token bucket or sliding window spec integrated into the client
8. **Retry policy** — exponential backoff, jitter, max attempts, retryable status codes (429, 5xx)
9. **Error types** — typed error hierarchy (WatsSenderAPIError, RateLimitError, AuthError, NetworkError)
10.**Logging** — structured logging interface (not implementation — just the interface the client expects)

## Differences from WassFlow's existing client

- No direct Supabase dependency — the client is a pure HTTP wrapper
- Add rate limiting (not the inline 5-second sleep — a proper token bucket)
- Add retry with exponential backoff
- Typed errors instead of generic `throw new Error()`
- Pluggable logger instead of `console.log`
- Media size validation before upload (5MB images, 50MB video, 16MB audio, 100MB docs — per WatsSender limits)

## Assets produced

- `specs/contracts/watsender-client-spec.md` — full specification with method docs, error types, retry logic
