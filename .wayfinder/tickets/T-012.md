# T-012: WhatsApp Automation Engine Specification (wa.*)

`wayfinder:task` | Parent: map.md

## Question

Produce the specification for the WhatsApp automation engine that lives in the `wa` schema. This includes the campaign engine for bulk sending, the sequence executor for automation workflows, and the send queue processor. Based on WassFlow's `campaign-engine.ts`, `sequence-executor.ts`, and queue processing code.

## Scope

### WA Campaign Engine (Bulk Send)
1. **Engine tick loop** — same pattern as WassFlow but org-scoped: iterate active campaigns per org, find pending events, check schedule, lock, enqueue
2. **Campaign event generation** — Broadcast mode (one product → all groups sequentially), Bulk Distribution (products × groups with randomization)
3. **Event scheduling** — delay randomization, wave management, jitter
4. **Overdue event re-spacing** — if events fall behind, shift forward (same as WassFlow's `reSpaceOverdueCampaignEvents`)
5. **Engine control** — start, pause, resume, cancel operations
6. **Completion detection** — `checkCampaignCompletion` logic
7. **Failed event retry** — rerun-failed logic with configurable max retries
8. **Differences from WassFlow** — multi-org scoping, no direct Supabase calls (repository pattern), Postgres function-based tick trigger via `pg_cron`

### Send Queue Processor
1. **Queue item selection** — oldest pending, per-session serialization, priority ordering
2. **Optimistic locking** — `eq('status', 'pending')` pattern for concurrency safety
3. **Session gap enforcement** — configurable minimum delay between sends per WA Poster session
4. **Presence simulation** — typing indicator before send (configurable)
5. **WatsSender execution** — call WatsSender client, capture result
6. **Success/failure handling** — update status, retry with exponential backoff, escalate to permanent failure
7. **Campaign event finalization** — on send success/failure, update campaign progress counters
8. **Sequence step progression** — for automation workflows, advance to next step
9. **Message status update** — mark inbound messages as completed/failed

### Sequence Executor (Automation Workflows)
1. **Trigger resolution** — match inbound message keyword against triggers (exact, contains, starts_with)
2. **Sequence execution** — load sequence steps, create send job, enqueue step messages
3. **Step progression** — send job tracks current_step/total_steps, advances on each successful send
4. **Delay handling** — delay_seconds per step, sleep between sends
5. **Error handling** — step failure → mark send job failed, stop sequence
6. **Variant selection** — if sequence has variant nodes, select based on rules

### Engine Scheduling
1. **pg_cron job** — runs every 60 seconds, triggers `wa.engine_tick()` Postgres function
2. **Queue processor** — runs continuously or triggered by `pg_cron` every 30 seconds
3. **Per-org fairness** — round-robin across orgs, prevent one org from starving others
4. **Engine health** — metrics endpoint (queue depth, processing rate, error rate)
