# T-009: Auth, RLS, and Multi-Tenancy Specification

`wayfinder:task` | Parent: map.md

## Question

Produce the complete authentication, authorization, and Row-Level Security specification. This is the backbone of multi-tenancy — every other spec depends on it being right.

## Scope

### Supabase Auth Setup
1. **User registration flow** — sign up → create Account → create Organization (or join existing)
2. **Login flow** — email/password, magic link, OAuth providers (which ones?)
3. **Session management** — JWT claims structure (`sub`, `org_id`, `role`), token refresh, session expiry
4. **Organization creation** — on first signup, auto-create Account + first Organization
5. **Organization invitation** — invite team members by email, accept/decline flow
6. **Organization switching** — users in multiple orgs switch context, JWT claims update

### JWT Claims Structure
```json
{
  "sub": "user-uuid",
  "email": "user@example.com",
  "org_id": "current-org-uuid",
  "role": "admin",
  "app_metadata": {
    "organizations": ["org-1-uuid", "org-2-uuid"],
    "default_org": "org-1-uuid"
  }
}
```

### RLS Policies
For EVERY table across both schemas (pm.* and wa.*):

1. **Tenant isolation policy** (applied to all tables with organization_id):
   ```sql
   CREATE POLICY tenant_isolation ON pm.products
   FOR ALL USING (organization_id = (auth.jwt() ->> 'org_id')::uuid);
   ```

2. **Tables that need special policies:**
   - `organizations` — only members can read, only admins can update
   - `organization_members` — only org members can read, only admins can manage
   - `notifications` — only the target user can read their own
   - `human_poster_working_hours` — user can read/write own, admins can read all in org
   - `action_logs` — all org members can read, only system can write (no direct user writes)

3. **Policy performance** — all policies must be simple equality checks (no subqueries, no joins)

### Auth Middleware
For Next.js API routes:
1. `withAuth(handler)` — extracts JWT, validates session, attaches `{ user, orgId, role }` to request
2. `withRole(handler, roles[])` — fails 403 if user's role not in allowed list
3. Service-to-service auth for webhook endpoints (API key validation)

### Frontend Auth
1. `useAuth()` hook — returns `{ user, orgId, role, organizations, switchOrg, signOut }`
2. `RequireAuth` component — redirects to login if unauthenticated
3. `RequireOrg` component — redirects to org creation if no org
4. Route protection — middleware.ts for page-level auth gating

### Security Considerations
- CSRF protection
- Rate limiting on auth endpoints (login, signup)
- Session invalidation on password change
- API key encryption for WA Posters (Supabase Vault or pgcrypto)
